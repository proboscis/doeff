## 生死判定の方法と再現手順

### 1. 定義数と構造の抽出

```sh

from pathlib import Path

PY
# 229 229
```

### 2. matcher 発火

各 rule の導入コミットを `git log --all -S "id: <RULE_ID>" -- .semgrep.yaml` で求め、`git diff-tree` で変更 path を列挙し、`git show <commit>^:<path>` から導入直前 corpus を一時領域へ構築した。matcher の生死と path の生死を混同しないため、履歴 corpus 走査では rule の `paths` だけを外し、元の matcher・language・severity は保持した。元の include/exclude は別途、現在の `git ls-files`（`tests/semgrep/fixtures/` を除外）との交差で検査した。

履歴で hit が得られない rule は、次の形で単一 rule を抽出し、rule ごとの専用 directory に最小 bad snippet を置いて確認した。実行時 Semgrep は `1.161.0`。bad snippet は一時領域だけに作り、リポジトリへ残していない。

```sh
RULE=doeff-no-return-control-primitives-in-handlers
uv run python - "$RULE" <<'PY'
import sys, yaml
from pathlib import Path
rule_id = sys.argv[1]
rules = yaml.safe_load(Path('.semgrep.yaml').read_text())['rules']
rule = next(r for r in rules if r['id'] == rule_id)
Path('/tmp/doeff-one-rule.yaml').write_text(yaml.safe_dump({'rules': [rule]}, sort_keys=False))
PY
# 元の paths に合う一時 tree と bad snippet を作り、その tree を cwd にする
(cd /tmp/doeff-semgrep-probe && semgrep --config /tmp/doeff-one-rule.yaml . --json --quiet --metrics off)
```

死亡疑いは、単に現コードが clean だったという意味ではない。規則が狙う構造を含む bad snippet と導入時履歴の双方に結果がなく、かつ Semgrep error/timeout ではないものだけを分類した。判定不能は0件だった。

### 3. 重複・包含関係

matcher の YAML 正規化比較で完全重複は1組だけだった: `vm-scope-no-mode-on-vm-struct` と `vm-g0-no-mode-register-on-vm` は language、regex、include path まで一致する。前者を DELETE候補とした。
部分包含として、`no-runresult-hasattr-duck-typing` と `cache-handler-no-result-duck-typing` は `hasattr(..., "is_ok")` を共有するが対象 path が異なる。`no-legacy-intercept-api` と `doeff-agentic-no-legacy-intercept` も pattern は重なるが後者は package 局所化であり、直ちに重複削除とはしない。VM の OCaml 5/G0/G1 群には語彙の包含が多いため、P1 fixture 作成時に同一 bad snippet への複数 hit を測って統合可否を再裁定する。
