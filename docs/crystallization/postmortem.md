### (b) クラスタ帰属

リビルド自体は相転移(godからの脱出)であり結合核の失敗ではない。ただしera内churnは **C2(継続の明示化)・C4(状態マージ意味論3連発: #4, #5, タスク分離 `01388ca4`)・C1(ハンドラ状態の置き場所の反復)** に集中。独立判断(#6, #7)もchurnしたが修正コストは線形(局所diff)で済んでいる。

### (c) 事前に固定できたはずのlaw

1. **状態合流law**(#4を固定): `Put(x,v); t←Spawn(q); Put(x,w); Join(t) ⊢ Get(x) ≡ w`(spawn後の親書き込みは子のjoinで保存)。last-write-winsとの差は紙上で判定可能だった — 実際にはISSUE-CORE-503系のバグとして実装後に発見。
2. **エラー経路の状態law**(#5を固定): `Try(Put(x,v); Fail) ; Get(x) ≡ v`(エラーは状態をロールバックしない)。逆を選ぶならtransactional law。どちらかを等式で先に宣言すべきだった。
3. **単一意味論law**(#3を固定): Awaitを含まないpについて `run_sync(p) ≡ run_async(p)`。Sync/Async二系統プロトコルはこの等式に最初から違反していた。

### (d) 相判定

**正当な発見相→脱出も適時**。[証言] 「早すぎなかった。さもなければ『ハンドラ』抽象を導入できなかった」— handler語彙の発見にはCESK機械が必要だった。era内churnのうちプロトコル反復は発見相の正当な反復。ただし**C4のマージ意味論churnはlaw未定義の症状**であり、機械がなくても等式で先に決められた(発見相でも「状態の規律」は雑にしてはならない、の実例)。

## 2. R2: Python CESK → Rust VM v1(2026-02-05〜02-11)

### (a) 覆った設計判断

| # | 覆った判断 | → 置換 | 根拠 | クラスタ |
|---|---|---|---|---|
| 1 | ホスト言語=Python(GIL保持、再帰限界、所有権強制不能) | Rust VM + PyO3、ジェネレータは葉のみ | `a4678266`(SPEC-008)、`c372d4b7`、`c41f1ca1` | 従属(主因はC2、下記b) |
| 2 | ハンドラ状態の置き場所: K内フレーム(WithHandlerFrame)とstore内ハンドラスタックの手動同期 | 二スタック分離(K=制御、handler_stack=境界)[agent読解: SPEC-008 ADR] | `16169354`(cesk_v3 SPEC-CESK-006)、`ec74d15f` | C1 |
| 3 | one-shotの強制: Pythonジェネレータの暗黙挙動(二重resumeはサイレント故障) | **ContId + consumed_continuations集合**による実行時検出 [agent読解: SPEC-008 ADR-6] | `c2ceac85` "enforce strict KPC and continuation invariants" | C2(**誤った点の選択** — R3で清算) |
| 4 | store = `dict[str, Any]` に全部入り(VM内部状態もユーザー状態も) | 型付きRustStore(state/env/log分離、VM内部は型付きフィールド) | `c372d4b7` [agent読解] | C4 |
| 5 | Free Monadフレーミング | Algebraic Effectsフレーミングへ転換 | `7b225c32`(2026-02-11)"reframe from Free Monad to Algebraic Effects" | 語彙の結晶化 |

**cesk_v3の検死**: R2直前の2日間(02-04〜05)、「unified K architecture」(全ハンドラ状態をKに統一、SPEC-CESK-006 v2)を実装(`16169354`→`01801973`)。これはC1の問題(K捕獲とハンドラ状態の同期)をPython内で解こうとした最後の試み。Rust VM specが出た02-05に放棄が事実上決まり、02-07に削除(`ec74d15f` "never integrated into the main runtime")。**C1の解として方向は正しかったが(捕獲が自然にハンドラ文脈を含む)、Pythonの制約(性能・所有権)は解けないため沈んだ** [推測を含む]。

### (b) クラスタ帰属

[証言] R2の決め手は「性能、コンパイラチェック、そろそろhandlerのような本物のエフェクト言語意味論を入れる頃合いだった」。つまり主因は **C2(所有権・線形性をコンパイラで強制したい)+ C1(cesk_v3が敗れた同期問題)+ 語彙の結晶化(handler意味論の本格導入)**。ホスト言語変更(#1)はそれらの従属判断であり独立の誤りではない。

重要な精緻化: **one-shotという意味論選択そのものは`10967cb2`(2026-01-12)から最後まで一度も覆っていない**。覆り続けたのはそれを表現・強制するメカニズム(暗黙→ContId+flag→Arc\<AtomicBool\>→move)。つまりC2の結合核は「one-shot vs multi-shot」の選択ではなく「**one-shotをスタック表現・所有権にどう束縛するか**」にある。

### (c) 事前に固定できたはずのlaw

1. **線形性law**(#3を正しく固定する): kは線形資源 — `Resume(k,v)`はkを消費し、二度目の使用は型エラー(実行時検出ではなく構成的に不可能)。この等式を「機構の要件」として先に置けば、ContId+consumedフラグという**観測ベースの強制**は最初から選外だった。R3後のSPEC-VM-021:282「At any point in time, a continuation exists in exactly ONE place」は、このlawの事後的な明文化である。
2. **deep handler等式**(#2/C1を固定): `handle h (E[perform e]) ≡ h(e, λv. handle h (E[v]))` — 捕獲された継続の再開はハンドラを再設置した文脈で行われる。文献の標準等式をR2時点で採用していれば「ハンドラ状態はKと別管理できるか」という問い自体が消える(捕獲範囲にハンドラが含まれることが定義になる)。
3. **恒等子不在law**: 継続に観測可能な同一性はない — `∀観測O: O(k)はchain(k)を経由して因数分解される`。これを置けばContIdは導入時点でlaw違反として弾けた。

### (d) 相判定

相転移自体(handler意味論の本格導入)は適時 [証言]。ただし**「意味論の結晶化」と「実装言語の結晶化」を同時に走らせ、前者が未完のまま後者が確定した**。one-shotの強制機構・継続の同一性・トレースの導出性が等式で固定されないままRustに入った結果、v1で補償機構が増殖する余地が残った。一方でRustの所有権型は最終的に正しい意味論(one-shot=move)の**発見装置**として機能した — 早すぎる結晶化の症状と、型システムによる反証の加速が同居した転移。

### (b) クラスタ帰属 — 結合核の実物標本

SPEC-VM-020自身が結合を明言している(:193-196):

> "DispatchId forces the existence of ProgramDispatch (to store it), DispatchTrace (to record what happened to it), TraceState (to accumulate traces), (...). **Remove DispatchId and all of this machinery dissolves.**"

さらに**部分修正の失敗例が文書化されている**(SPEC-VM-021:6-11): PR #371はContIdだけ除去しようとして `Arc<AtomicBool>` 共有フラグを導入し「**made the architecture WORSE**」— 結合核の一部だけ動かすと別の補償機構が生えるという、仮説Xの「結合核の誤りは非線形に高くつく」のミニチュア再演。8項目すべてが「継続の所有権が一意でない」という単一の違反から派生しており、**8件の独立判断ではなく1つの結合核(C2)+その派生(C5)** である。

[証言] これらの補償機構は明示的設計判断ではなく「**実装中になし崩しに増殖**」した。つまり結合核の失敗様式は「誤った判断」ではなく「**判断の不在**」— 閾値未満のモデルが局所制約(one-shotを動かす・トレースを出す・スケジューラを動かす)を各個撃破するたび、大域制約(単一所有)の不在を恒等子・レジストリ・フラグ・側表で埋めた。H1a(大域制約を自発保持しない)の機構的確認。

### トリガーの認定(オーナー記憶は不明瞭のため証拠ベース)

SPEC-VM-020(3/17)直前の2週間は消火活動の連続: stale traceバグ(`7973cd4e` "stale handler dispatch classification in traceback"、TRACE-STALE-TELL `73900415`、トレース整形バグ `b6e3049a`/`c818d328`)、scheduler hang/デッドロック(`430f8503` ExternalPromise waiter map、`34e558fb`/`b62b9c6d` 再現テスト群、`dd2a05a8` O(N²))、継続チェーンバグ(`38150405` "Fix resumed continuation caller chain")。側表除去の試行も先行(`9ad9ba50` 3/13 "Remove VM dispatch side table")。**認定: 補償機構群のバグ圧が閾値を超え、構造認識(SPEC-VM-020)に至った**。ブランチ名 rebuild/drop-stale-trace はこの文脈。

### (c) 事前に固定できたはずのlaw

1. **単一所在不変条件**: 「継続は任意の時点でちょうど1箇所に存在する」(SPEC-VM-021:282 — R3後に明文化されたが、R2時点で書けた)
2. **恒等子不在law**: 継続・ディスパッチに観測可能な同一性はない(同一性=head fiberポインタ)
3. **トレース導出law**: `traceback(vm) ≡ F(fiber_chain(vm))` — トレースはVM状態の純関数であり蓄積物ではない。「staleなトレース」はこのlawの否定形として最初から検出できた
4. **スケジューラ・ライブラリlaw**(C5を固定): scheduler ≡ {Spawn, Wait, Gather, …}上の再帰的WithHandler。「コアプリミティブだけでスケジューラが書ける」はプリミティブ集合の完全性の判定式でもある(Task 4へ接続)

### (d) 相判定

**正当な再結晶化**。自前発明(DispatchId等)をやめ、確定済みの外部意味論(OCaml 5)を輸入した。v1の失敗は「結晶化が早すぎた」のではなく「**結晶化の参照点を持たずに実装が走った**」こと — R2で文献等式を採用していればv1の形そのものが成立しなかった。なおOCaml 5整合後、アーキテクチャ違反テスト13本で判断が機械的に固定された(= 判断の償却の成功例)。

## 4. 横断パターン(同型3回以上 → 一般化)

1. **補償機構の増殖則**: 継続の所有権が一意でないとき、補償として恒等子(ContId/DispatchId)・レジストリ・共有フラグ・蓄積トレースが増殖する。所有権を一意化(move)するとすべて消える。(R2導入 → v1肥大 → R3清算の3点で確認)
2. **意味論選択は安定、束縛機構が覆る**: one-shot(2026-01-12)、handler抽象(R1)、エフェクト=データ(初期)という意味論レベルの選択は一度も覆っていない。リビルドで覆るのは常に「その意味論を状態表現・所有権にどう束縛するか」。→ 結晶化すべき最優先対象は意味論と表現の**束縛規則**(=law)である。
3. **C4は全時代で出血**: 状態×継続の意味論は R1(マージlaw書き換え・エラー時ロールバック廃止・タスク分離)、R2(store分割)、R3(writer_log/global_stateの置き場所)と毎回手当てされ、現在も「global_stateは捕獲時コピーなし・全fiber共有」が**コードの形でしか存在しない**(packages/doeff-vm-core/src/vm/var_store.rs)。one-shot前提なら整合するが、lawとして未宣言(Task 3/4で固定対象)。
4. **判断の不在が結合核を壊す**: 3回とも、結合核をまたぐ明示判断の誤りではなく、局所修正の複利が大域制約を侵食した([証言] なし崩し増殖)。

## 5. 仮説Xの判定

**判定: 真**(ただし下記の精緻化付き)。

- R3はほぼ純粋な結合核リビルド(8項目が単一違反の派生、スペック自身が結合を明言、部分修正PR #371の失敗まで文書化)。
- R2の主因はC2(所有権強制)+C1(捕獲とハンドラ状態の同期)であり、ホスト言語変更すら結合核の従属判断。
- R1は相転移であってリビルドの病理ではないが、era内churnはC1/C2/C4に集中。
- 対照群: 独立判断(API命名、Runtime→関数、Result/Maybe値化、エフェクト整理)も多数覆ったが、**いずれも局所diffで修正されリビルドを誘発していない**。コストの非線形性は結合核に固有。

**精緻化1**: 結合核の失敗様式は「誤った選択」より「**選択の不在**」が支配的。相互拘束する判断は、誰も明示的に選ばないとき、局所修正の積み重ねが暗黙に(そして非整合に)選んでしまう。→ ルーティング規則への含意: 結合核は「フロンティアが選ぶ」だけでなく「**選択肢が等式として明文化されるまで実装を着手させない**」ゲートが要る。

**精緻化2**: H1bの「9割の壁」の説明として: 失敗は結合核に集中し(X真)、かつその失敗はモデルの局所最適化の複利として静かに進行するため、レビューで個々のdiffを見ても捕まらない(各局所修正は単体では正しい)。検出可能な単位は「不変条件の違反」であり、これがTask 3(check_invariants)の存在理由。
